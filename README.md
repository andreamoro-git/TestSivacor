# README — SIVACOR test package

A minimal Stata replication package built to exercise the
[SIVACOR](https://docs.sivacor.org/) automated reproducibility service. It uses
two simulated raw data sources — **one treated as confidential** — merges them,
runs a regression, and produces one table and one figure. It also demonstrates
`.sivacorignore`, which excludes non-redistributable data from the certified
output while keeping it available during the run.

## Overview

The pipeline is driven by a single controller script, **`main.do`** — this is
the file to designate as the *main do-file* when submitting to SIVACOR. SIVACOR
sets the working directory to the folder containing `main.do`, which the code
captures as `$rootdir` and uses to build every other path relatively. There are
no absolute or hardcoded paths.

## Data availability and provenance statement

All data are **simulated** by the authors — there are no real human subjects,
surveys, or third-party sources. The data are constructed by `make_data.do`
(Stata's `runiform()` / `rnormal()` with a fixed seed).

To exercise SIVACOR's handling of non-redistributable data, this package
**treats one of the two sources as confidential**:

- `data/raw/firm_registry.dta` — public firm characteristics; redistributable.
- `data/raw-confidential/firm_survey.dta` — firm survey outcomes, treated as
  **confidential** (as if under a Data Use Agreement). It is listed in
  `.sivacorignore`, so it is present during the run but **excluded from the
  redistributed replication package / TRO output**.

Because the survey file is simulated, it is included here so the package can be
run; in a real submission the confidential source would be obtained separately
and would never be redistributed.

## Computational requirements

### Software
- **Stata** (code written to `version 14`; runs on Stata 14 or any later
  version). Only built-in commands are used — `regress`, `twoway`,
  `graph export`, `merge`, `file write` — so **no SSC/ado packages need to be
  installed** and the package requires no network access.

### Memory and runtime
Trivial: ~500 simulated observations. Runs in a few seconds on any machine;
well under a megabyte of memory.

## Folder structure

```
TestSivacor/
├── main.do                       <- MAIN controller (designate this in SIVACOR)
├── make_data.do                  <- author-only data construction (NOT in the pipeline)
├── README.md
├── .sivacorignore                <- excludes confidential data from the output
├── code/
│   ├── 02_merge.do               <- merge registry + survey on firm_id
│   └── 03_analysis.do            <- regression + table + figure
├── data/
│   ├── raw/                      <- firm_registry.dta (public)
│   ├── raw-confidential/         <- firm_survey.dta (confidential; in .sivacorignore)
│   └── derived/                  <- analysis_sample.dta (generated by main.do)
└── output/
    ├── tables/                   <- table1_regression.txt (generated)
    └── figures/                  <- figure1_revenue_capital.png (generated)
```

The `data/derived/` and `output/` subfolders are recreated by `main.do` if absent.

## Instructions to replicators

The raw data are **provided** with the package (the confidential survey file is
included so the code can run; `.sivacorignore` keeps it out of the redistributed
output). The reproducible pipeline does **not** rebuild the data.

1. Designate `main.do` as the main do-file.
2. Run it. It executes `02_merge.do` then `03_analysis.do`: merge → analysis,
   producing the table and figure.

There is nothing to download and no path to edit. `make_data.do` (at the package
root) is the author-only script that originally built the raw data; replicators
do not need to run it.

## List of outputs

| Output | Program | File |
|--------|---------|------|
| Table 1 — OLS of log revenue on inputs | `code/03_analysis.do` | `output/tables/table1_regression.txt` |
| Figure 1 — Revenue vs. capital scatter + fit | `code/03_analysis.do` | `output/figures/figure1_revenue_capital.png` |

## What the analysis does

`log_revenue` is regressed on `log_capital`, `log_employment`, and industry
fixed effects. Because the data are simulated from
`log_revenue = 2 + 0.5·log_capital + 0.4·log_employment + industry shifter + noise`,
the estimated coefficients should recover those values up to sampling error.
